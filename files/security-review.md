백엔드 코드 품질 + 보안 심층 평가

  잘하고 있는 것

  SSaC→핸들러 충실도가 높다. 모든 핸들러가 SSaC 스펙의 @auth → @get → @state → @call → @put 순서를 정확히 따르고, early return 패턴도 일관적이다.
  구체적으로:

  - 테넌트 격리 완벽 — 모든 조회에 FindByIDAndOrg(id, currentUser.OrgID) 사용. 타 조직 리소스 접근 시 404 반환 (IDOR 방어)
  - 상태 머신 정확 — Mermaid stateDiagram과 transitions 맵이 1:1 대응, self-loop(active→ExecuteWorkflow→active) 포함
  - 트랜잭션 일관성 — 상태 변경 핸들러에 BeginTx / defer Rollback / Commit 패턴 일관 적용
  - 인증/인가 분리 — middleware(JWT 검증) → handler(@auth OPA 체크) 2단 구조
  - SQL Injection 방어 — 모든 쿼리 parameterized ($1, $2), 문자열 직접 연결 없음
  - 서명 알고리즘 검증 — VerifyToken에서 *jwt.SigningMethodHMAC 타입 체크 (alg substitution 공격 방어)

  총평

  백엔드는 구조적으로 잘 생성되어 있다. SSaC 스펙 충실도, 테넌트 격리, SQL 파라미터화, JWT alg 검증 등 기본기가 갖춰져 있다. 하지만:

  1. DISABLE_STATE_CHECK 환경변수는 제거하거나 빌드 태그로 전환해야 한다 — 런타임 우회는 위험
  2. RefreshToken의 하드코딩 시크릿은 fullend codegen 버그로 보인다 — IssueToken과 동일 패턴이어야 함
  3. JWT 기본 시크릿 "secret" 폴백은 제거하고 필수값으로 만들어야 한다

  S1은 fullend의 상태머신 codegen이 디버그용 코드를 산출물에 남긴 것이므로, fullend 자체의 버그로 보고해야 할 사안이다.
