# 3-Layer PreLLM + MapReduce 컨텍스트 아키텍처

## 핵심 원리

컨텍스트를 누적하지 않는다. 매 턴 필요한 것만 탐색해서 조립한다.
기존 100k 풀컨텍스트 대신 턴당 5-10k 상수 비용.

## 상시 유지 데이터

### 키워드맵 (~500토큰)
매 턴 종료 시 키워드 추출하여 갱신. 최근 100-1000개 유지.
```
hurl,capture,BUG004,canResolvePathParams,ErrStatus,scanXxx,...
```

### 대화 세션 JSONL
원본 대화 전체. 요약하지 않음. grep 대상.

### whyso 캐시
파일별 변경 이력. `whyso history --grep` 으로 키워드 필터링 가능.

### 프로젝트 키워드맵 (`_map.md`)
whyso map 산출물. 코드베이스 함수/엔드포인트 전체 목록.

## 턴 처리 흐름

### 0. 분류 (규칙 기반 또는 haiku)
사용자 프롬프트가 질문/요청/확인 중 무엇인지 판별.

### 1. PreLLM 1차 — 키워드 선택 (haiku)
- 입력: 사용자 프롬프트 (~100) + 키워드맵 (~500) = ~600토큰
- 출력: 관련 키워드 JSON 배열 (~50토큰)
- 역할: 탐색 범위 결정

### 2. Grep — 수집 (프로그래밍적, 토큰 0)
키워드로 아래 소스를 병렬 탐색:
- 대화 세션 JSONL → 관련 턴 발췌
- 발견된 파일명으로 `whyso history --grep <keyword>` → 결정 이력
- 프로젝트 `_map.md`에서 관련 함수/파일 위치 특정

### 3. PreLLM 2차 — MapReduce 필터링 (haiku, 병렬)

**Map**: 발견된 파일마다 독립적으로 관련성 판단
- 입력: 질문 (~100) + 파일 발췌 (~500) = ~600토큰/파일
- 출력: `{ relevant: true/false, read: "라인범위" }` (~30토큰)
- 10파일 병렬 → 총 ~6,300토큰, 레이턴시는 1회분

**Reduce**: 통과한 파일만 수집
- 프로그래밍적, 토큰 0
- 지정된 라인 범위로 파일 발췌 (Read)
- whyso history 결과 병합

### 4. MainLLM — 답변 (sonnet/opus)
- 입력: 조립된 컨텍스트 (~5,000토큰)
- 출력: 실제 답변
- 풀컨텍스트 불필요. 탐색 파이프라인이 정제한 정보만 수신.

## 비용 비교

| | 기존 (풀컨텍스트) | 3-Layer PreLLM |
|---|---|---|
| 1턴 | 10k | ~8k |
| 10턴 | 50k | ~8k |
| 30턴 | 95k→컴팩트→20k | ~8k |
| 100턴 누적 | ~500만 토큰 | ~80만 토큰 |
| 대화 길이 영향 | 비례 증가 | 상수 |
| 정보 손실 | 컴팩트 시 요약 손실 | 없음 (원본 grep) |

## 탐색 품질이 전부

이 아키텍처의 성능 = 탐색 파이프라인 품질.

- 키워드맵 품질: 매 턴 추출 정확도
- PreLLM 1차: 키워드 선택 정확도
- Grep 범위: 대화 + whyso + 코드맵 커버리지
- PreLLM 2차 Map: 파일별 관련성 판단 정확도

fullend 프로젝트는 SSOT가 이미 정제된 인덱스 역할을 하므로 탐색 품질이 구조적으로 보장됨.

## 확장

- PreLLM 단계는 가변: 1-4회. 충분한 컨텍스트를 찾을 때까지 반복 가능.
- 키워드 추출은 규칙 기반으로도 가능 (함수명, 파일명, 에러코드 패턴 매칭)
- Flowave 에이전트에 탑재 시 채팅 기반 개발 환경에서 턴당 비용 일정하게 유지
